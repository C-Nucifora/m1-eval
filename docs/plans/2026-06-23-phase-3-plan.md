# m1-eval Phase 3 Implementation Plan — Log import + counterfactual replay

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the **headline feature** — log-driven **counterfactual replay**. Import a recorded MoTeC run (CSV always; binary `.ld` behind a cargo feature) into channel time-series, resample onto the engine's tick grid, treat every logged channel as ground truth, let the user **override** one or more channels (constant or expression), re-evaluate only the **downstream dependency cone** of each override, leave everything else at logged values, and emit both the new series and a **per-channel diff vs the logged series**. The load-bearing invariant: a no-op override reproduces the logged series within tolerance.

**Architecture:** Unchanged tree-walking interpreter over `m1-core`'s CST + `m1-typecheck`'s `Project`. Phase 3 adds:
1. a **log module** (`src/log.rs`) producing a format-agnostic `Log` (channel → time-series) from CSV or `.ld`;
2. a **`.ld` reader** (`src/log/ld.rs`, behind feature `ld`) built on the MIT crate `motec-i2`;
3. a **downstream-cone runner** in `runner.rs` — the *mirror image* of the existing upstream `build_cone`: from each override channel, walk the writer-map **forward** (readers-of, transitively) to the set of functions that must be recomputed, topo-ordered writer-before-reader;
4. a **counterfactual orchestration** on `Engine` (`load_log` / `override_channel` / `run_counterfactual`) returning a `Trace` + a `Diff`.

No new core evaluator, value store, or trace primitive is required — Phase 3 reuses `Env`/`StateStore`/`tick_loop`/`exec_script`/`summary::io_sets`/`Trace`. The override + cone machinery is a new *runner mode* + a new *diff type*, not a new interpreter.

**Tech Stack:** Rust (edition 2024, rust-version 1.88). One new **optional** dependency: `motec-i2 = "0.2"` (MIT, GPL-compatible), gated behind feature `ld`. CSV import needs no new dependency (the existing `scenario::split_csv_row` + `Scenario::load_csv` CSV machinery is the basis).

## Global Constraints

- **Dependency pins (unchanged):** `m1-core` = `v0.12.0`, `m1-typecheck` = `v0.36.0`. No upstream change is required or permitted — everything is built inside `m1-eval` against the existing public API.
- **License:** `GPL-3.0-or-later`. Every new source file carries `// SPDX-License-Identifier: GPL-3.0-or-later`. The one new dependency (`motec-i2`, MIT) is GPL-compatible; the `ld` feature and its provenance are documented in `README.md` + the module header.
- **Clean-room `.ld` provenance — NEVER reverse-engineer MoTeC software.** The `.ld` reader is built **only** on independently-published, license-compatible work: the MIT crate `motec-i2` (parses+writes `.ld`/`.ldx`) and the GPL `ldparser` format docs. We parse an independently-documented *file format* operating on the user's *own* telemetry. No MoTeC software is decompiled; no MoTeC manual text, calibration, firmware, or sample log is copied or committed.
- **No proprietary data committed.** Committed CI fixtures are **synthetic**: a hand-written CSV and a tiny synthetic `.ld` generated by `motec-i2`'s writer at test time. Any test against the real 173 MB EV-M1 `.ld` corpus is **env-gated** (`M1_EVAL_LOG_DIR`, mirroring the existing `M1_EVAL_EVM1_DIR` pattern) and off the default CI path.
- **Fail loud, never silently wrong:** a malformed log, a non-numeric cell, an override of a channel with no in-project reader (nothing downstream to recompute), or an `.ld` opened without the `ld` feature → an `EvalError`. No guessed value, ever.
- **M1 identifiers may contain spaces** (`Drive State`, `AMK Inverter Boot State`) — only ever split channel paths on `.`, never whitespace. Log channel names map to canonical paths via the existing `ident::classify` and a documented name-mapping pass (`.ld` short names ↔ M1 channel paths).
- **Determinism:** fixed tick grid + explicit `dt`; logs are resampled by zero-order hold (the established `InputSeries::sample` rule). Same log + same overrides ⇒ identical `Trace` + `Diff`.
- **No toolchain types leak past the public API:** the new public surface is `Log`, `Diff`, `Override`, and the new `Engine` methods — all `m1-eval` types. `motec-i2`'s types stay inside `src/log/ld.rs`.
- **Every milestone is independently `cargo test`-greenable** (with `--all-features` for the `ld` tasks). No milestone leaves the tree red.

## Grounded facts (verified against the real `.ld` + the `motec-i2` source + the m1-eval checkout)

### Verified `.ld` binary structure (hexdump of `S1_#19134_20260525_135516.ld`, 37 MB, under `M1_EVAL_LOG_DIR`)

- **Header @ 0x0000** is a fixed offset/pointer table of little-endian `u32`s: bytes `40 00 00 00` (`0x40` = pointer to channel-metadata region / first-channel address), `e2 06 00 00`, `53 32 03 00`, `21 c4 06 00` (channel-data pointer), then reserved/zero words, and `1e e3 02 00` @ 0x24 (a meta/event pointer). Confirms the documented "header offset table" layout.
- **Device/identity strings** (fixed-width, NUL-padded, in the header block): `M1` / `M150` device type @ ~0x4a/board area; ASCII date `25/05/2026` @ 0x5c, time `13:55:16` @ 0x7c, driver `adam` @ 0xa0, vehicle id `EV25` @ 0xde, venue `lakeside` @ 0x160, session `afternoon`.
- **Metadata key/value strings** (in the first 64 KB): `project.name`, `version.{major,minor,build}`, `device.boardtype`, `device.serialnum`, `Summary.Firmware`, `Summary.Firmware Version` (`01.00.0161`), `Summary.Hardware` (`M150`), `Summary.System Version`, `Summary.Firmware Licence`, `Summary.VehicleId` (`EV25`), `modified.utc`, `pkg.calibrations`, `migrated_from.data_hash`, and an `_objs.prop` XML blob naming tables (`Control.Limiting.DC Current`, etc.). These exactly match the spec's licensing-assessment evidence.
- **`.ldx` sidecar** is a separate small XML file (lap/beacon markers + `Total Laps`/`Fastest Time` details) — **NOT** channel data. `motec-i2`'s reader does not touch it; Phase 3 does **not** import `.ldx` (out of scope — markers, not values).

### `motec-i2` crate API (verified — crates.io v0.2.0, MIT, edition 2021, July 2024; single dep `byteorder ^1.5`; pure `std`)

- **Reader:** `LDReader::new(source: &mut S)` over a seekable byte source (`std::io::Cursor<Vec<u8>>` or `File`); `read_header() -> I2Result<Header>`; `read_channels() -> I2Result<Vec<ChannelMetadata>>`; `channel_data(&ChannelMetadata) -> I2Result<Vec<Sample>>` (also `read_event`/`read_venue`/`read_vehicle`).
- **`Header`** fields: `channel_meta_ptr: u32`, `channel_data_ptr: u32`, `event_ptr: u32`, `device_serial: u32`, `device_type: String`, `device_version: u16`, `num_channels: u32`, `date_string`, `time_string`, `driver`, `vehicleid`, `venue`, `session`, `short_comment: String`.
- **`ChannelMetadata`** fields: `prev_addr`/`next_addr`/`data_addr`/`data_count: u32`, `datatype: Datatype`, `sample_rate: u16`, `offset: u16`, `mul: u16`, `scale: u16`, `dec_places: i16`, `name: String`, `short_name: String`, `unit: String`.
- **`Sample`** enum: `I16(i16) | I32(i32) | F32(f32)`. **`Datatype`** enum: `Beacon16 | Beacon32 | I16 | I32 | F16 | F32 | Invalid`.
- **Scaling (load-bearing):** `channel_data` returns **RAW** samples; physical value = decode via `offset/mul/scale/dec_places` (the crate exposes a `decode_f64(channel)` path in its tests). **Phase 3 must apply this scaling** to get engineering units. **Time** of sample `i` = `i / sample_rate` seconds (the reader stores `sample_rate` Hz but does no time math).
- **Writer (enables the synthetic `.ld` fixture):** `LDWriter::new(sink: &mut S, header: Header)` → `.with_channel(ChannelMetadata, Vec<Sample>)` (chainable) → `.write() -> I2Result<()>`. A `Cursor<Vec<u8>>` sink + a couple of small channels lets us generate a tiny committed-CI synthetic `.ld` at test time with no proprietary bytes.

### Existing m1-eval seams Phase 3 reuses (read from the checkout, not memory)

- **`Scenario::load_csv(&mut self, csv: &str)`** (`src/scenario.rs`): already parses a `time`-first-column CSV into per-channel `InputKind::Series` and merges into `self.inputs`, replacing same-named inputs. Phase 3's **CSV schema is exactly this** (documented below); the log path reuses `split_csv_row` and the resampling rule rather than inventing a second CSV reader.
- **`InputSeries { channel, kind }` + `InputKind::{Const, Series(Vec<(f64, Value)>)}` + `InputSeries::sample(t)`** (zero-order hold) — the canonical "drive a channel from a time series" type. A `Log` is, in effect, a `Vec<InputSeries>` plus metadata; resampling onto the tick grid is `sample(t)` per tick, identical to how the tick loop already seeds scenario inputs.
- **`summary::io_sets(script, project, group) -> IoSets { writes, reads }`** — the read/write sets per function. The existing **upstream** `build_cone` (in `runner.rs`) builds a `writer: channel→script` map from `io.writes` and walks **reads backward**. Phase 3 adds the **downstream** walk: from an override channel, find functions that **read** it (forward edge), recompute them and everything they feed, transitively. Same `IoSets`, same `topo_order`, opposite direction.
- **`runner::tick_loop(loaded, scenario, &[ScheduledRated], base_rate_hz)`** — the shared deterministic loop seeding inputs/overrides per tick, running each gated function, holding unwritten channels (zero-order hold), recording the `Trace`. The counterfactual run is a `tick_loop` over the **downstream cone schedule** with the **log seeding every logged channel** as input and the **overrides** layered on top — no new loop.
- **`scenario.overrides: Vec<InputSeries>`** + the tick loop's existing "seed inputs, then layer overrides" pass — overrides already exist as a concept and are already applied last. Phase 3 wires log + override into this same precedence (calibration < scenario < **log** < override) and adds the downstream-only recompute.
- **`Trace { time, channels: BTreeMap<String,Vec<Value>>, exprs, external }`** + `to_json`/`to_csv` — unchanged; the counterfactual produces a normal `Trace`. The `Diff` is computed *between* two `Trace`s (logged-baseline vs counterfactual) or between the logged `Log` and the counterfactual `Trace`.
- **`Engine`** facade (`src/engine.rs`): `load` / `run` / `coverage`. Phase 3 adds `load_log` / `override_channel` / `run_counterfactual` without leaking toolchain types.
- **`EvalError`** variants (`src/error.rs`): reuse `UnsupportedConstruct` / `UnresolvedSymbol` / `TypeError` / `MissingCalibration`-style fail-loud; add a log-specific variant only if the existing ones do not read cleanly.

### Verified CSV-log schema (documented, the always-unencumbered path)

A Phase-3 log CSV is the existing `Scenario::load_csv` shape, formalised:
- **Row 1 (header):** `time,<channel name>,<channel name>,…`. The first column header MUST be `time` (case-insensitive). Channel headers are M1 channel paths verbatim (spaces allowed, e.g. `Engine Speed`); RFC-4180 quoting per `split_csv_row`.
- **Optional row 2 (units):** if the second row's first cell is non-numeric (e.g. `s,rpm,km/h`), it is treated as a **units header** and skipped for values (matching real i2 exports, which carry a units row). A numeric first cell means no units row. (This is the one documented extension over `load_csv`, needed for real i2 CSVs.)
- **Data rows:** `t_seconds,value,value,…`. `time` is seconds, ascending; numeric cells parse to `Value::Float`; empty cells hold the previous value (the column simply gets no keyframe at that `t`). Non-numeric value cells fail loud.
- **Resampling:** at tick `t`, each channel is sampled by zero-order hold (`InputSeries::sample`) — the established, deterministic rule.

## Testing Strategy

- **Synthetic fixtures only, no proprietary data:** the committed CSV fixture and the synthetic `.ld` (generated by `motec-i2`'s writer at test time) carry no MoTeC bytes. The `counterfactual/` project fixture is hand-authored synthetic (Sensor→Mid→Out chain + an unrelated channel). Real `.ld`/EV-M1 tests are env-gated (`M1_EVAL_LOG_DIR`) and `#[ignore]`-by-default.
- **The no-op invariant is the load-bearing gate (Task 9):** an identity override must reproduce the logged series within tolerance and yield an empty changed-channel set — a whole-pipeline differential check that exercises log import, resampling, cone-building, override application, recompute, and diff together.
- **Downstream-cone correctness (Task 4):** explicit tests that the override's cone is exactly the forward reachability set (A,B recomputed; unrelated C untouched), the mirror of the existing upstream `build_cone` tests; and the fail-loud "override has no reader" case.
- **Scaling correctness (Task 11):** at least one hand-derived `.ld` sample decoded through `offset/mul/scale/dec_places` to a known engineering value — raw-vs-scaled is a real correctness risk (`motec-i2` returns RAW samples).
- **Feature-flag hygiene:** `cargo test` (no `ld`) and `cargo test --features ld` both green at every `.ld` task; the `.ld` code is fully `#[cfg(feature="ld")]`-gated, and the no-feature `load_log(.ld)` path fails loud with a clear "rebuild with --features ld" message.
- **Fail-loud preserved:** malformed CSV (missing `time`, non-numeric value), override of a non-read channel, `.ld` `Datatype::Invalid`, `.ld` without the feature — each has a test provoking the `EvalError`.
- **Regression:** the existing 287-test suite stays green after every milestone; the `scenario::load_csv` refactor (Task 2) must not change its observable behaviour (its existing tests are the guard).
- **Determinism:** same log + same overrides ⇒ byte-identical `Trace` + `Diff` (an `insta` snapshot on the counterfactual `Trace` + `Diff` of the synthetic fixture guards scheduling/resampling/diff regressions).

## `.ld` strategy — DECISION: depend on `motec-i2` as an optional crate (not vendor, not hand-roll)

**Decision:** add `motec-i2 = { version = "0.2", optional = true }` and gate all `.ld` code behind a cargo feature `ld`. Use it for **both** reading (`LDReader`) and writing the **synthetic CI fixture** (`LDWriter`).

**Rationale (licensing + engineering):**
- **License-compatible + clean-room.** `motec-i2` is **MIT** (GPL-3.0-compatible) and is an independent, published reverse-engineering of the *file format* (not MoTeC software). Building on it means we parse an independently-documented format operating on the user's own telemetry, and **never reverse-engineer MoTeC software** — sidestepping the usual EULA "no reverse engineering of our software" clause, exactly as the spec's licensing assessment concludes (low risk + guardrails). The GPL `ldparser` docs corroborate the layout (clean-room provenance) and are a cross-check, not a code source.
- **Depend, don't vendor.** It is published on **crates.io (v0.2.0, July 2024)** with a single small dependency (`byteorder`), pure `std`, edition 2021 — a clean, pinnable dependency. Depending keeps provenance crisp (the licence travels with the crate), avoids carrying a copy of someone else's code in our GPL tree, and lets `cargo` enforce the version pin. (If upstream ever goes unmaintained or a fix is needed, vendoring-with-attribution remains a fallback — but there is no reason to vendor now.)
- **Don't hand-roll.** A from-scratch `.ld` parser would re-derive the format we'd still be reading from `ldparser`/`motec-i2` anyway, adding format-fidelity risk for no licensing benefit (we'd still be parsing the same format). The writer (for synthetic fixtures) would be pure extra surface. `motec-i2` gives reader **and** writer, and the writer is what makes a committed, no-proprietary-bytes CI fixture possible.
- **Guardrails (from the spec, enforced here):** CSV is the always-available, unencumbered path (Milestone P3-A, no feature needed); `.ld` is **behind the `ld` feature** with documented clean-room provenance; **no MoTeC calibration/firmware/manual/sample-log bytes are committed** (synthetic fixture only; real `.ld` env-gated); the user confirms their specific MoTeC software-licence terms before shipping the `.ld` reader.
- **Caveat (carried from the spec):** the signed MoTeC software EULA was not on disk (it lives in a Parallels Windows VM), so this is "low risk + verify," not certified clearance.

## Out of scope (do NOT build here)

- **`.ldx` import** — the sidecar is lap/beacon markers + summary details (XML), not channel values; `motec-i2`'s reader ignores it and so do we. Lap-aware analysis is a later concern.
- **Faithful `.ld` *writing* of real runs** — the writer is used only to generate the tiny synthetic CI fixture; we do not export real traces back to `.ld`.
- **`.m1dbc` CAN decode / faithful CAN-bus replay** — logged CAN channels can be fed as ordinary log channels; bit-level DBC decode is not required here.
- **LSP hover/inlay** — Phase 4.
- **Non-zero-order-hold resampling** (linear/spline interpolation of logs) — zero-order hold is the documented, deterministic rule; richer interpolation is a fidelity follow-up, not a Phase-3 gate.
- **Multi-lap / time-reset logs** (`.ldx` beacon-driven lap segmentation) — single monotonic time axis only in v1.
