# m1-eval Phase 1.5 + Phase 2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the real-data builtin gaps the EV-M1 `--coverage` run surfaces (**Phase 1.5**), then build the **whole-project multi-rate scheduler** — the faithful mini-ECU (**Phase 2**). Phase 1.5 makes a real EV-M1 control script runnable end-to-end (pure `Calculate` overloads, enum `.AsInteger` conversions, project-object method routing, inline user-function calls). Phase 2 adds a `RunMode::WholeProject` that runs every scheduled function each base tick at its own rate (500/200/50/10/2 Hz) in dependency-then-rate order over a fixed duration, producing a deterministic `Trace`.

**Architecture:** Unchanged from Phase 1 — a tree-walking interpreter over `m1-core`'s CST using `m1-typecheck`'s `Project`/`SymbolTable`/`resolve`/`ValueType`. Phase 1.5 extends the builtin **dispatch** seam (`builtins::dispatch`, case 5: `dispatch_object_method`) and the **call** seam (`expr::eval_call`) — no new core. Phase 2 adds a third runner alongside the single-function and dependency-cone runners in `runner.rs`, reusing `summary::io_sets` for ordering and the existing `Env`/`StateStore`/tick loop for execution.

**Tech Stack:** Rust (edition 2024, rust-version 1.88). No new dependencies — `m1-core` `v0.12.0` and `m1-typecheck` `v0.36.0` (already pinned) expose every API this plan needs.

### Execution rate (Phase 2 scheduling)

- **`m1_typecheck::symbols::Symbol::call_rate_hz: Option<f64>`** — the function's execution rate in Hz, derived by `m1-typecheck` from the `.m1prj` `<Props SelectedTrigger="…Events.On <N>Hz">` pointing at a `BuiltIn.EventKernel` clock. `None` for symbols with no trigger, `On Startup` functions, and triggers that are unresolvable `$(…)` parameter references.
- The rate of the function backing a script: `project.symbols().get(&fn_path).and_then(|s| s.call_rate_hz)`, where `fn_path = project.function_symbol_for_script(&script.name)`. This is the **exact** pattern `m1-typecheck`'s `schedule.rs::script_io` uses (`let rate_hz = project.symbols().get(&fn_path).and_then(|s| s.call_rate_hz);`).
- `m1-typecheck`'s `schedule.rs::ScriptIo` (which carries `fn_path`, `rate_hz`, `writes`, `reads`, `calls`) is **private** — not exported. m1-eval already derives its own equivalent via `summary::io_sets`; Phase 2 derives rate the same way (the one-liner above), not by reaching into `schedule.rs`.
- To **enumerate scheduled functions**: iterate `loaded.scripts`; for each, `function_symbol_for_script(&name)` gives the symbol path, then `.call_rate_hz`. A script whose function has `call_rate_hz = Some(hz)` is a periodically scheduled function at `hz`; `None` means unscheduled (startup/event) — excluded from the periodic schedule.
- `Symbol::log_rate_hz: Option<f64>` exists (channel default log rate) but is **not** the execution rate — do not use it for scheduling.

### User-function call evaluation (Phase 1.5 + Phase 2)

- A user function is a `Symbol` of `SymbolKind::Function` (`BuiltIn.FuncUser` / `BuiltIn.FuncUserParam` / `BuiltIn.FuncGenerated` / `BuiltIn.CalFuncUser`) or `SymbolKind::Method` (`BuiltIn.MethodUser`). Confirmed mapping in `symbols/m1prj/mod.rs::from_classname`. In EV-M1, `Torque Vectoring.Update` and `Slip Control.Update` are `BuiltIn.FuncUserParam` → `SymbolKind::Function`.
- The **backing script** ↔ symbol: `project.function_symbol_for_script(file_name) -> Option<String>` maps a script file to its function symbol path (explicit `Filename=` match first, else the `Root.<stem>` convention, e.g. `Control.Slip Control.Update.m1scr` → `Root.Control.Slip Control.Update`). To go symbol→script, build the reverse: for each `ParsedScript`, compare `function_symbol_for_script(&s.name)` to the callee path.
- `Symbol::in_params: Option<Vec<(String, ValueType)>>` — the declared `<Signature><Params>` (name → type, in order). `None` when the component declares no signature (then args are unchecked/opaque).
- `Symbol::return_type: Option<ValueType>` — inferred from the body's `Out = <expr>` assignments by `Project::infer_return_types` (already called in `loader::load`). The **return value** at runtime is whatever the body assigns to `Out`.
- `In.<Param>` resolves via `m1_typecheck::resolve::resolve` to `Resolution::Local(ty)` **only** when `scope.fn_symbol`'s `in_params` carries that param name; `m1-eval`'s `ident::classify` maps that to `Target::Local("In.<param>")`. So binding a call argument = storing it under the local key `In.<param>` for the callee's frame.
- `Out` resolves to `Resolution::Opaque` (function-local return object, never a project symbol) → `ident::classify` currently returns `Target::Unresolved`. A dedicated `Env` "out slot" is therefore required for both writing (`Out = …` in the body) and reading the return value.

### Enum member → integer (`.AsInteger`)

- `m1_typecheck::symbols::EnumType { name: String, members: Vec<(String, i64)>, default: Option<String>, open: bool }`. The **`i64` in `members` is the member's integer value** — `ContainerOrder` for project-local enums (`symbols/m1prj/component.rs::parse_enums`), the documented `EnumMember.value` for builtin/firmware enums (registered via `add_builtin_enum`). This is exactly what `.AsInteger()` must return.
- Access: `SymbolTable::enum_by_name(name) -> Option<EnumId>`, `SymbolTable::enum_type(id) -> &EnumType`, `SymbolTable::enum_has_member(id, member) -> bool`. `EnumId = usize`.
- `.AsInteger` is **not** a stored symbol and **not** a library method — it is an `objectMethod` in the intrinsics (`intrinsics::get().object_method("AsInteger")` is non-empty), and through `resolve` a path ending in `.AsInteger` is `Resolution::Opaque` (an accessor on a resolved prefix). The evaluator must special-case it in dispatch — verified `objectMethods` includes `AsInteger, AsString, Constrain, GetScaled, GetUnscheduled, Lookup, Receive, Remaining, Reset, Set, Start, Stop, Validate`.
- In EV-M1 it is always called **with empty parens**: `… .AsInteger()` (a `CallExpression` whose callee is a `MemberExpression` ending in `AsInteger`). Two source forms, both resolved through `EnumType.members`:
  1. **Enum-type-qualified member literal** (compile-time constant): `AMK Inverter Boot State.System Ready.AsInteger()`, `Drive State.Idle.AsInteger()`, `Sensor Diagnostic Enumeration.OK.AsInteger()`. Here the object path is `<EnumTypeName>.<Member>` — resolve the enum type by `enum_by_name`, find `(member, value)` in its `members`, return `Value::Int(value)`.
  2. **Value-holding source** (runtime): `Control.Drive State.AsInteger()`, `Driveline.Inverter.Rear Left.Boot State.AsInteger()`, `Discrete.Diagnostic.AsInteger()`. Here the object resolves to a `Channel` (enum-typed, e.g. `Boot State`) or a `GroupCompound` value-compound (e.g. `Control.Drive State`, whose enum value lives on its `.Value` child). Read the current `Value::Enum { id, member }` and convert via `enum_type(id).members`.

### Project-object methods (routing)

- `SymbolKind` from classname (`symbols/m1prj/mod.rs::from_classname`): `BuiltIn.Channel*`→`Channel`; `BuiltIn.Parameter*`→`Parameter`; `BuiltIn.Constant*`→`Constant`; `BuiltIn.FuncUser*`/`CalFuncUser*`/`FuncGenerated`→`Function`; `BuiltIn.MethodUser`/`EventKernel`→`Method`; `BuiltIn.Table*`→`Table`; `BuiltIn.GroupCompound`→`Group`; `BuiltIn.Reference`→`Reference`; any non-`BuiltIn.*` class (e.g. `MoTeC Input.Sensor`, `_IOMethod.*`, DBC-sourced `BuiltIn.CAN.*`)→`Object`.
- The EV-M1 `--coverage` unsupported list (verified by running `m1-eval --coverage` against `UQR-EV/01.00.0166/Project.m1prj` + `parameters.m1cfg`) partitions exactly into the categories Phase 1.5 closes:
  - **Pure `Calculate`:** `Calculate.Absolute`, `Calculate.Average`, `Calculate.NAN` (used); plus intrinsics also list `Infinity, InverseCos, InverseSin, InverseTan, IsFinite, MaximumFloat` (implement for completeness). Also `Math.atan2` appears (calibration-only `Math` object used in a script — route to the existing `Calculate.InverseTan2` math or fail-loud as calibration-only; see Task 3).
  - **Enum `.AsInteger`:** `AMK Inverter Boot State.{System Ready,HV Activated}.AsInteger`, `Drive State.{Idle,Precharging}.AsInteger`, `Sensor Diagnostic Enumeration.OK.AsInteger` (literal form); `Control.Drive State.AsInteger`, `Driveline.Inverter.{Rear,Front} {Left,Right}.Boot State.AsInteger`, `Discrete.Diagnostic.AsInteger` (value form).
  - **DBC CAN message-object methods** (object class `BuiltIn.CAN.*`, sourced from a `.m1dbc`): `Balls3EV25.DashVals.{Tx,TxInitialise,TxOpen}`, `Balls3EV25.DashVals.Aux Switch.SetBit`, `Balls3EV25.DashVals.Inverter Error.SetUnsignedInteger`, `Balls3EV25.Init`, `IZZE DBC.*.{GetScaled,Receive}`, `SBG DBC.*.{GetScaled,Receive}` → **externally-driven IO stubs** (like the `CanComms` library object).
  - **GroupCompound `.Update()`:** `Driveline.Inverter.{…}.Service Bits.Update` (`Service Bits` is `BuiltIn.GroupCompound`) → externally-driven IO stub (a CAN service-bits push).
  - **Channel `.Set(…)`:** `Driveline.Accumulator.Precharge State.Set` (`Precharge State` is `BuiltIn.Channel`) → a **channel write** (imperative setter; the same write the schedule analysis already treats `Chan.Set*` as).
  - **Project Timer / Output / other object methods:** `Timer.{Start,Remaining}`, `Startup Delay.{Start,Remaining}` (project `Timer` objects — `stateful::timer` already handles these at runtime, but `classify_builtin` reports them unsupported; reconcile in Task 5), `Output.SetState`, `Vehicle.TSAL Buzzer.Buzze` → documented stubs / fail-loud per object kind.
  - **User functions:** `Slip Control.Update`, `Torque Vectoring.Update` → inline user-function evaluation (Task 6).
- A method call written as `A.B.C.Method(args)` reaches `expr::eval_call` with a `MemberExpression` callee; `eval_call` flattens the object to `A.B.C` and calls `builtins::dispatch("A.B.C", "Method", …)`. The routing fork for **all** of the above therefore lives in `dispatch` (case 5, `dispatch_object_method`) and, for user functions, in `eval_call`/`dispatch` — these are the only two seams Phase 1.5 touches.

### Existing seams Phase 2 reuses

- `runner::run(loaded, scenario)` dispatches on `scenario.mode` (`RunMode::Function` | `RunMode::Cone`); `tick_loop(loaded, scenario, &[Scheduled])` is the shared deterministic loop. `Scheduled { script, group, fn_symbol }`. Phase 2 adds a `RunMode::WholeProject` arm that builds the full ordered schedule and calls `tick_loop` — **with a per-tick gate** so each function only runs on the ticks its rate divisor selects (see Task 9; `tick_loop` is generalised to accept a rate per scheduled function).
- `summary::io_sets(script, project, group) -> IoSets { writes, reads }` (canonical paths) already exists and is what the cone runner orders by — Phase 2 reuses it verbatim for the dependency ordering layer.
- `Env` (value store + `enter_function`/`leave_function`, statics keyed by `fn_symbol`), `StateStore` (per-`CallSite` stateful slots), `Trace` (channel columns + per-expr sink + `mark_external`) are unchanged and shared across all three runners.

## Milestone P2-B — Rate-gated tick loop

### Task 13: Base tick + per-function rate divisors

**Files:** Modify `src/runner.rs` (generalise `tick_loop`).

The base tick rate is `scenario.base_rate_hz` (default to the fastest scheduled rate when the scenario does not pin one — **decision:** if `base_rate_hz` is absent or `0`, use `max(rate_hz)` over the schedule; document this). Each function runs every `divisor = round(base_rate_hz / rate_hz)` ticks; its per-run `dt` is `1 / rate_hz` (the time since *its* last run), **not** the base `dt`.

- [ ] **Step 1 (failing test):** with `base_rate_hz = 100`, a 100 Hz function runs every tick (divisor 1) and a 50 Hz function runs every other tick (divisor 2). - [ ] **Step 2 (implement):** generalise `tick_loop` to take `&[ScheduledRated]` and a `base_rate_hz`. Precompute each function's `divisor` and per-run `dt`. The shared `Env`/`StateStore`/`Trace` carry values between runs, so a function not run this tick leaves its last-written channels in `env` (zero-order hold) — the trace's per-tick recording then naturally repeats the held value. Keep the `single-function` and `cone` runners working by passing them through the generalised loop with `divisor = 1` and `dt = base dt` (they have no per-function rate).

## Testing Strategy

- **Unit-per-gap, hand-derived expectations:** every new `Calculate` overload, the `.AsInteger` conversion (both forms, proving `ContainerOrder`/`value` is used — not the ordinal index), the `.Set` setter, each IO stub, and the user-function frame mechanics has an explicit-number test.
- **Fixture realism, synthetic data:** new fixtures (`enums/`, `multirate/`, `userfn/`) are hand-authored synthetic projects — no proprietary EV-M1 content committed. The real EV-M1 acceptance gates (Task 9, Task 18) are path-env-gated and off the default CI path.
- **Coverage parity:** after each Phase-1.5 milestone, `classify_builtin`/the coverage report must agree with what `dispatch` actually does — a unit test asserts the just-added `(object, method)` pairs are reported `Supported`/`Stubbed` (never silently `Unsupported` while runtime succeeds, nor vice-versa).
- **Determinism + snapshot (Phase 2):** the whole-project run is the strongest determinism test — same scenario twice must be byte-identical, with an `insta` snapshot guarding the schedule/`dt`.
- **Fail-loud preserved:** new error paths (unknown enum member, `.Set` on a non-channel, a recursive user call, a `.AsInteger` on a non-enum) each have a test that provokes the `EvalError`.
- **Regression:** the full existing 202-test suite stays green after every milestone; the one intentional change is the `unknown_mode_fails_loud` scenario test (Task 10), which moves its negative case off `"whole-project"` now that that mode is valid.
